# Python DDD Constitution

## 核心原则

### I. 分层边界优先
所有实现必须遵守 `interfaces -> application -> domain` 与 `infrastructure -> application/domain` 的单向依赖。领域层禁止感知 FastAPI、SQLAlchemy、Pydantic、Session、HTTP 协议与外部 SDK 细节。任何功能变更都必须先判断业务规则应归属领域层还是应用层，禁止在路由、Schema、ORM Model 中堆叠核心业务逻辑。

### II. 领域模型承载业务规则
业务不变量、实体行为、领域异常、仓储端口定义优先落在领域层；应用层只负责用例编排、事务边界和 DTO 转换；基础设施层只负责持久化映射与端口实现；接口层只负责收参、调用用例、响应映射。若实现让核心层直接依赖数据库或 HTTP 细节，则视为设计不合格。

### III. 规格先于实现
新增功能、缺陷修复和重构默认遵循 `specify -> plan -> tasks -> implement` 流程。规格文档先描述业务目标、边界条件、验收标准与影响模块，再进入实现。未澄清业务上下文、领域对象、规则归属与数据库变更前，不得直接从路由层开始编码。

### IV. 测试门禁不可跳过
新增功能、缺陷修复、重构、数据库映射调整、接口行为调整后，必须补充与本次改动对应的测试，并执行 `pytest`。若测试失败，必须先判断是否由本次改动引起；由本次改动引起的失败必须修复。没有测试和 `pytest` 结果支撑，不得宣称任务完成。

### V. 迁移与适配器一致性
凡涉及表结构变更，必须同步更新 ORM Model 与 Alembic 迁移，不允许只改其一。仓储实现负责 ORM Model 与领域实体之间的映射，禁止把 ORM 对象直接暴露给应用层、接口层或 API 响应。需要事务时，由应用层定义事务边界，由基础设施层提供执行支撑。

## 补充约束

- 所有新增注释、说明、文档注释必须使用简体中文。
- 标识符命名使用英文，避免把中文混入变量名、函数名、类名和模块名。
- 优先做最小必要改动，但不能以“少改”为理由破坏分层与依赖方向。
- 发现现有代码存在明显的分层越界时，应在本次改动范围内做最小必要纠偏，而不是继续叠加错误实现。
- API 层必须使用 Pydantic Schema 作为请求和响应模型，不直接暴露 ORM 对象或领域实体。

## 开发流程

1. 先识别需求所属模块、涉及的领域对象、规则归属与是否需要数据库迁移。
2. 先设计核心层：按需补充领域实体、值对象、仓储接口、领域服务、应用层 DTO、Use Case。
3. 再补适配器：按需实现仓储、ORM Model、路由、Schema 与依赖注入。
4. 如有表结构变更，补充 Alembic 迁移并校验模型与迁移一致。
5. 为本次改动补充测试，最后执行 `pytest` 验证回归。

## 治理规则

- 本宪章在 spec-kit 工作流中作为默认执行约束，与仓库根目录 `AGENTS.md` 保持一致；若两者出现表述差异，以更严格的一方为准。
- 任一规格、计划、任务拆解和实现方案都必须显式检查分层边界、测试要求和迁移要求。
- 若用户请求与本宪章冲突，优先保证核心边界、已有功能稳定性与可测试性，并在交付说明中写明权衡点。
