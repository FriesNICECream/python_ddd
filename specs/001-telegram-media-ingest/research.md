# 研究记录：Telegram 影视资源采集

## 决策 1：Telegram 接入方式采用“用户会话读取”适配器建模

- Decision: 规划中将 Telegram 读取能力定义为基础设施层的“用户会话型消息拉取适配器”，由应用层通过端口调用。
- Rationale: 需求明确是“我加入了几个群聊想定时获取内容”，核心前提是读取个人账号已加入群聊的历史或增量消息，而不是单纯接收机器人被动事件。把该能力建模为读取适配器，可以让核心流程不依赖具体 SDK，并保留未来替换客户端库的空间。
- Alternatives considered: 仅基于 Bot API 的机器人接入。放弃原因是它天然更偏向机器人身份与被授权群的事件处理，不适合作为“个人已加入群聊”采集方案的默认前提。

## 决策 2：去重采用三层保护，而不是单表唯一键

- Decision: 使用“原始消息幂等键”“资源主记录匹配规则”“网盘链接唯一性”三层保护共同完成去重。
- Rationale: 同一影片可能由不同群、不同消息、多次转发反复出现。若只依赖资源名称唯一键，容易误合并；若只依赖消息唯一键，又无法把不同消息下的同一资源汇总为一条主记录。三层保护可以兼顾去重精度与增量沉淀。
- Alternatives considered: 仅使用资源名称唯一约束。放弃原因是名称存在轻微差异、别名、季数附加文本和修饰词，单一唯一键不能承担完整业务规则。

## 决策 3：基础刮削信息先采用可扩展字段容器

- Decision: 首期只沉淀资源基础信息与待整理状态，基础刮削字段采用可扩展结构保存，不在 v1 固化完整整理规则。
- Rationale: 你已明确“整理逻辑后续完成，先拿到 Telegram 信息后决定”。因此当前设计应优先支撑采集沉淀与后续演进，而不是过早把刮削流程写死。
- Alternatives considered: 首期即定义完整刮削字段全集与整理流程。放弃原因是当前规则尚未明确，过早固化会导致迁移和模型重复调整。

## 决策 4：定时采集与手动采集复用同一应用用例

- Decision: 定时任务只负责触发应用层采集用例，不单独实现一套入库逻辑。
- Rationale: 采集核心业务是“从来源拉增量消息并沉淀资源”，无论由调度器还是人工触发，本质都是同一个业务动作。统一入口更利于测试和幂等。
- Alternatives considered: 为调度任务和 API 手动触发分别实现两套流程。放弃原因是会造成逻辑漂移与测试重复。
