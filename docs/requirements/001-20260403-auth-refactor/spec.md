# 登录与注册重构 Spec

---

## 1. 基本信息

- 需求名称：登录与注册功能重构
- 需求编号：AUTH-REFactor-001
- 提出人：用户
- 日期：2026-04-03
- 关联模块：`user` / `auth`

---

## 2. 目标

将当前仅支持“注册用户 + 按 ID 查询用户”的示例能力，重构为具备基本认证语义的“注册 + 登录”能力，并确保实现符合仓库既有约束：

- 保持 DDD 与六边形架构分层正确
- 让登录与注册的核心规则进入核心层，而不是落在路由或 ORM 中
- 对外暴露清晰的 API 输入输出
- 为后续扩展令牌、权限、会话等能力保留演进空间

---

## 3. 当前现状

当前仓库仅存在 `user` 示例模块，现状如下：

- 已有注册接口：`POST /api/users`
- 已有按 ID 查询接口：`GET /api/users/{user_id}`
- 用户领域对象仅包含：
  - `id`
  - `email`
  - `full_name`
  - `created_at`
- 当前注册流程不包含密码
- 当前不存在登录用例
- 当前不存在密码校验、凭证校验、令牌签发等认证能力
- 当前数据库模型 `users` 表中没有密码相关字段

结论：

- 当前系统本质上不是“登录/注册系统”，只是一个最小用户示例
- 若要实现真正的登录与注册，必须补充认证相关核心模型与持久化结构

## 4. 范围

### 4.1 包含范围

- 重构现有注册能力，使其支持认证所需的最小字段
- 新增登录能力
- 明确登录与注册的核心业务规则
- 明确登录与注册的分层落位
- 如有必要，新增认证相关应用用例、仓储接口和基础设施实现
- 补充本次改动对应测试

### 4.2 不包含范围

- 权限系统
- 角色系统
- 刷新令牌
- 第三方登录
- 短信/邮箱验证码
- 找回密码
- 邮件激活

---

## 5. 核心业务规则

1. 注册必须校验邮箱唯一性。
2. 注册必须提供可用于后续登录的凭证信息。
3. 登录必须基于用户标识与凭证校验。
4. 密码或凭证的明文不能直接以领域外泄方式传播。
5. 认证失败时应返回清晰但不过度暴露细节的失败结果。
6. 注册与登录规则不能直接写在路由或 ORM Model 中。

### Domain

- 定义用户/认证核心对象及业务规则

### Application

- 编排注册用例与登录用例
- 处理 DTO 转换
- 协调用户仓储与凭证服务端口

### Infrastructure

- 实现仓储
- 实现密码哈希与校验适配器
- 调整 ORM Model 与数据库映射

### Interfaces

- 提供注册与登录 API
- 处理请求体与响应体转换
- 映射应用异常到 HTTP 响应

## 9. 风险点

1. 若直接在现有 `user` 示例上堆密码逻辑，可能让用户上下文与认证职责混杂。
2. 若先写接口再反推领域模型，容易把认证规则写进路由层。
3. 若不明确登录成功返回语义，后续接口和用例会频繁返工。

## 10. 假设 / 待确认项

- 假设：
  - 首版采用邮箱 + 密码作为登录方式
  - 首版注册时需要 `email`、`full_name`、`password`
  - 首版密码采用哈希存储，不存明文
- 待确认：
  - 登录成功是否只返回用户信息，还是返回访问令牌

  - 是否允许保留现有 `GET /api/users/{user_id}` 不变
