# AGENTS.md

本文件用于约束 Codex 在本仓库中的行为。目标不是提供“参考建议”，而是提供一套**高约束、强模板、防误用、可稳定执行**的开发规范。

除非用户明确要求突破本规范，否则 Codex 必须把本文件视为当前仓库的最高本地约束。

---

## 1. 仓库定位

本仓库是基于以下技术栈的后端模板：

- `FastAPI`
- `SQLAlchemy 2.0`
- `PostgreSQL`
- `Alembic`

当前目录结构：

```text
app/
  domain/          # 领域层
  application/     # 应用层
  infrastructure/  # 基础设施层
  interfaces/      # 接口层
alembic/           # 数据库迁移
```

本仓库采用：

- `DDD`
- `六边形架构`
- `分层依赖单向流动`

Codex 的默认目标不是“快速把功能塞进去”，而是：

1. 不破坏架构边界
2. 不破坏已有功能
3. 让后续需求还能继续稳定演进

---

## 2. 强制总原则

以下规则不是建议，是**必须遵守**：

1. 必须按 `domain -> application -> infrastructure / interfaces` 的职责拆分代码。
2. 必须优先把业务规则放入领域层，而不是路由层、ORM 层、Schema 层。
3. 必须保证依赖方向正确，禁止内层依赖外层。
4. 必须让每次改动可被测试验证，不能以“理论上没问题”代替验证。
5. 必须控制变更范围，优先做最小必要改动，但不能为了最小改动牺牲分层正确性。
6. 必须默认新增注释、说明、文档为**简体中文**。
7. 必须避免“临时方案永久化”，发现明显破坏架构的旧代码时，应在本次改动范围内做最小必要纠偏。
8. 必须默认所有 spec-kit 产物、任务说明、计划说明、交付说明使用**中文为主**；允许保留必要英文专业术语，但应配合中文语境表达。

## 3. 依赖方向

唯一允许的主依赖方向：

```text
interfaces -> application -> domain
infrastructure -> domain
infrastructure -> application
```

禁止出现：

```text
domain -> application
domain -> infrastructure
domain -> interfaces
application -> interfaces
application -> infrastructure(具体实现)
```

### 3.1 依赖规则解释

- `domain` 是核心中的核心，必须纯净。
- `application` 只依赖领域模型和领域定义的端口，不依赖具体技术实现。
- `infrastructure` 是适配器，实现仓储、数据库、外部服务等细节。
- `interfaces` 是输入适配器，负责 HTTP 请求和响应协议转换。

### 3.2 Domain 层禁止依赖

`app/domain/` 中**禁止导入**：

- `fastapi`
- `sqlalchemy`
- `pydantic`
- 数据库 `Session`
- HTTP Request / Response
- Alembic
- ORM Model

### 3.3 Application 层禁止依赖

`app/application/` 中**禁止直接依赖**：

- SQLAlchemy ORM Model
- 路由函数
- FastAPI Request / Response
- 具体仓储实现类
- 直接 SQL 细节

### 3.4 Interfaces 层禁止行为

`app/interfaces/` 中**禁止**：

- 直接写数据库 CRUD 细节
- 直接操作 `Session` 完成业务流程
- 直接实现核心业务规则
- 直接返回 ORM 对象作为 API 响应

### 3.5 Infrastructure 层禁止行为

`app/infrastructure/` 中**禁止**：

- 定义核心业务规则
- 替代领域实体承担业务行为
- 把 ORM Model 当作领域模型到处传递

## 4. 各层职责模板

### 4.1 Domain 领域层

路径：`app/domain/`

只允许放：

- 实体 `entities.py`
- 值对象
- 聚合根
- 领域服务 `services.py`
- 仓储接口 `repositories.py`
- 领域异常 `exceptions.py`
- 纯业务规则

必须满足：

1. 领域对象表达业务含义，不表达 HTTP 或数据库细节。
2. 业务不变量优先放在领域层。
3. 仓储只定义接口，不写实现。
4. 能用领域对象约束的规则，不要丢给接口层兜底。

### 4.2 Application 应用层

路径：`app/application/`

只允许放：

- 用例 `use_cases.py`
- DTO `dto.py`
- 命令/查询处理逻辑
- 事务协调
- 对领域对象与仓储接口的编排

必须满足：

1. 一个用例只做一个清晰的业务动作。
2. 应用层负责编排，不负责承载底层技术细节。
3. 如果出现大量业务判断，应考虑下沉到领域层。
4. 应用层依赖仓储接口，不依赖仓储实现。

### 4.3 Infrastructure 基础设施层

路径：`app/infrastructure/`

只允许放：

- ORM Model
- 仓储实现
- 数据库 Session 管理
- 第三方服务适配器
- 配置化技术细节

必须满足：

1. 实现领域层定义的端口。
2. 负责 ORM 与领域对象之间的映射。
3. 技术细节集中，不向核心层泄漏。

### 4.4 Interfaces 接口层

路径：`app/interfaces/`

只允许放：

- 路由
- 请求/响应 Schema
- 参数解析
- API 错误映射
- 调用应用层用例

必须满足：

1. 路由函数保持薄。
2. 只做收参、调用、响应转换。
3. HTTP 协议层异常在接口层转换。
4. 不在此层完成真正的业务决策。

## 5. 六边形架构执行规则

在本仓库中：

- `domain + application` 是核心
- `interfaces + infrastructure` 是适配器

Codex 必须按以下方式思考和落地：

1. 先确定核心模型，再接适配器。
2. 先定义端口，再写实现。
3. 先保证核心可测试，再接 HTTP/DB。
4. 替换 Web 框架或数据库时，核心层应尽量不需要改动。

如果某个实现让核心层感知到：

- SQLAlchemy
- FastAPI
- PostgreSQL 方言
- 外部 SDK 细节

则默认判定为设计不合格，应重构后再继续。

## 6. 新功能开发强模板

当用户提出一个新功能时，Codex 必须按以下顺序执行，不得跳步。

### 第 1 步：识别业务上下文

必须回答清楚：

- 该需求属于哪个模块
- 涉及哪些领域对象
- 新规则属于领域层还是应用层
- 是否涉及数据库结构变更

如果这些问题未厘清，禁止直接从 `routes.py` 开始写。

### 第 2 步：先设计核心层

按需新增：

- 领域实体
- 值对象
- 仓储接口
- 领域服务
- 应用层 DTO
- 应用层 Use Case

### 第 3 步：再补适配器

按需新增：

- 仓储实现
- ORM Model 调整
- FastAPI Schema
- FastAPI 路由
- 依赖注入接线

### 第 4 步：补迁移

如果表结构变更，必须：

1. 补充 Alembic 迁移
2. 不允许只改 Model 不改迁移
3. 不允许只改数据库不留迁移文件

### 第 5 步：补测试并执行 pytest

必须执行：

1. 先补足与改动对应的测试
2. 再执行 `pytest`
3. 确认没有破坏旧功能

如果 `pytest` 未通过，默认任务**未完成**。

## 7. 新模块目录模板

新增业务上下文，例如 `order`，默认采用以下结构：

```text
app/
  domain/
    order/
      __init__.py
      entities.py
      repositories.py
      services.py
      exceptions.py
  application/
    order/
      __init__.py
      dto.py
      use_cases.py
  infrastructure/
    repositories/
      order_repository_sqlalchemy.py
    db/
      models.py
  interfaces/
    api/
      schemas.py
      routes.py
```

执行规则：

1. 小功能可以复用现有文件，但不能把文件堆成“大杂烩”。
2. 一旦单文件开始同时承担多种职责，必须拆分。
3. 不允许把领域规则放到 `schemas.py` 或 `routes.py` 中凑合实现。

## 8. 编码规范

### 8.1 语言与注释

必须遵守：

1. 所有新增注释、说明、文档注释使用**简体中文**。
2. 变量名、函数名、类名、模块名使用英文。
3. 注释解释“为什么”，不要解释显而易见的“做了什么”。
4. 允许保留必要英文专业术语，但注释、文档、spec-kit 产物整体表达必须以中文为主，不得默认输出整段英文说明。

### 8.2 类型与函数

必须遵守：

1. 新增函数尽量补齐类型标注。
2. 应用层 DTO、仓储接口、用例返回值必须有清晰类型。
3. 避免超长函数。
4. 避免在一个函数中混合：
   - 参数校验
   - 业务判断
   - 持久化
   - 响应序列化

### 8.3 复用与重复

必须遵守：

1. 禁止复制粘贴业务逻辑到多个层。
2. 禁止为了省事把通用业务规则塞到工具函数里到处调用。
3. 复用优先通过领域服务、用例、仓储接口完成。

---

## 9. 数据库与 ORM 强约束

必须遵守：

1. 表结构变更必须通过 Alembic。
2. ORM Model 只负责持久化映射。
3. 领域实体不等于 ORM Model。
4. 仓储实现负责 ORM 与领域对象之间转换。
5. 不允许把 `Session` 传入领域层。
6. 不允许在路由层直接写事务性持久化流程。

如果需要事务：

- 事务边界在应用层定义
- 由基础设施层提供事务执行支撑

---

## 10. API 开发强约束

必须遵守：

1. 请求体与响应体使用 Pydantic Schema。
2. API 层不直接暴露 ORM 对象。
3. API 层不直接暴露领域实体。
4. 领域异常不要直接写成 HTTP 异常。
5. HTTP 状态码和响应格式映射放在接口层。

路由函数默认只允许出现以下三类动作：

1. 解析参数
2. 调用应用层用例
3. 返回响应

出现第 4 类以上复杂行为时，默认说明分层开始失控，应回退整理。

---

## 11. 测试与回归保护

这是本文件的**强制要求**，不是建议。

### 11.1 强制测试规则

Codex 在以下场景后必须执行 `pytest`：

1. 新增功能
2. 修复缺陷
3. 重构代码
4. 调整数据库映射
5. 调整接口行为

### 11.2 完成定义

只有同时满足以下条件，任务才算完成：

1. 代码实现完成
2. 必要测试已补充
3. `pytest` 已执行
4. `pytest` 结果通过，或用户明确接受当前失败状态

缺少任一项，都不算真正完成。

### 11.3 测试优先级

优先补：

1. 领域层纯业务规则测试
2. 应用层用例测试
3. 接口层集成测试

### 11.4 失败处理规则

如果 `pytest` 失败：

1. 先定位失败是否由本次改动引起
2. 若由本次改动引起，必须修复
3. 若为历史失败，也要在交付说明中明确指出
4. 不允许把失败结果一笔带过

### 11.5 无测试时的处理

如果仓库现有测试不足：

1. 不能以“项目没测试”为理由跳过验证
2. 必须至少补本次改动相关测试
3. 再执行 `pytest`

---

## 12. 防误用清单

Codex 在本仓库中**禁止**以下行为：

1. 在 `routes.py` 中直接写数据库 CRUD 流程。
2. 在 `domain` 中导入 `fastapi`、`sqlalchemy`、`pydantic`。
3. 用 ORM Model 直接充当领域实体。
4. 让应用层直接依赖具体仓储实现。
5. 为了赶进度跳过分层。
6. 在 Schema 中写业务规则。
7. 在 ORM Model 中堆业务逻辑。
8. 不补测试就宣称功能完成。
9. 未执行 `pytest` 就宣称没有破坏旧功能。
10. 发现架构边界明显错误仍继续在错误位置叠代码。

---

## 13. 标准产出模板

Codex 在实现需求时，默认应产出以下内容：

### 13.1 代码层面

- 正确分层的代码变更
- 必要的测试
- 必要的迁移文件

### 13.2 说明层面

最终说明应至少覆盖：

1. 改了什么
2. 为什么这样分层
3. 是否补了测试
4. 是否执行了 `pytest`
5. `pytest` 的结果如何
6. 是否存在遗留风险

如果没有执行 `pytest`，必须明确说明原因，不能省略。

---

## 14. Codex 执行口令

Codex 在本仓库中处理需求时，应默认按下面的内部检查顺序自检：

1. 这段业务规则是否放错层了？
2. 这段依赖方向是否反了？
3. 这段逻辑是否应该进入领域层？
4. 这段实现是否把 ORM 或 HTTP 细节泄漏进核心层？
5. 这次改动是否补了测试？
6. 这次改动是否执行了 `pytest`？
7. 如果现在交付，是否有证据证明没有破坏旧功能？

只要其中任意一项回答不成立，就不能草率结束任务。

---

## 15. 最终判定标准

一次合格的 Codex 产出，必须同时满足：

- 分层正确
- 依赖正确
- 业务规则位置正确
- 注释与说明为简体中文
- 测试覆盖本次改动
- 已执行 `pytest`
- 没有把“暂时可跑”当成“设计正确”

若用户需求与本文件冲突，优先选择：

1. 不破坏核心边界
2. 不破坏已有功能
3. 可被测试验证

如果必须做权衡，Codex 应明确说明权衡点，而不是静默做出会污染架构的实现。
